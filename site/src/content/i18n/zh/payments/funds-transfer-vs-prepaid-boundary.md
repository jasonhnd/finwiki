# 日本的资金转账与预付费边界

## 维基路线

该条目位于 [[payments/INDEX|payments index]] 下。使用 [[payments/card-acquiring-japan-stack|Japan card acquiring stack]] 来读取相邻上下文，使用 [[fintech/japan-stablecoin-regulatory-landscape|日本 Stablecoin 法制度の三層構造（JPYC・USDC・Project Pax）]] 来读取更广泛的系统边界。

## 长话短说

实际边界是**可转移性**。资金转账服务处理银行外的外汇交易，并将价值转移给另一个人。预付支付工具主要是储值，用于向发行人或参与商户付款；它并不是被设计为开放的个人对个人的汇款渠道。

这种区别对于代码支付、钱包、存储余额、游戏积分、过境货币、商家礼品价值和积分产品至关重要。产品品牌不可靠：同一个应用程序可以在不同的法律层中包含资金转账余额、预付金额、积分和卡关联支付。

## 边界测试

| 问题 | 资金转移信号 | 预付费信号 |
|---|---|---|
| 用户可以向另一个人发送价值吗？ | 是的，特别是如果该服务接受汇款指令。 | 通常不会；余额主要用于用户自付。 |
| 该服务是否在银行外进行外汇交易？ | 这指向资金转移登记。 | 通常不是核心功能。 |
| 该价值是否用于从发行机构或参与商户购买商品/服务？ | 可能，但其本身还不够。 | 核心用例。 |
| 仪器终止时，未使用的价值是否受到保护并可退款？ | 用户资金保护取决于资金转移类别。 | 根据《支付服务法》规则终止使用时，适用预付费退款程序。 |
| 余额是大额汇款还是小额汇款？ | 类别事项：I、II、III 类资金转账业务。 | 预付规则和发行人义务比汇款限额更重要。 |

## 资金转移类别

FSA 材料描述了 2021  资金转移后的三个类别：

| 类别 | 功能 | 主要制约因素 |
|---|---|---|
| 第一类资金转账业务 | 超过 JPY 1  百万日元的高额转账。 | 授权;接受没有具体汇款指示的资金通常受到限制。 |
| 第二类资金划转业务 | 转账金额为 1  百万日元或更少。 | 登记;可能需要减少多余的无关余额。 |
| 第三类资金划转业务 | 日元50,000 或以下的小额转账。 | 登记;收款人账户余额暂时不能超过日元50,000 ，在有条件的情况下允许隔离存款保障。 |

这种分类很重要，因为对用户来说看起来像“余额”的钱包可能会受到不同的监管，具体取决于它是否执行汇款以及可以发送或持有多少价值。

## 预付费层

FSA 将预付支付工具解释为一个广泛的类别，包括礼品券、礼品卡、磁性/IC 预付卡和基于服务器的预付工具。预付费发行人可能需要备案或注册，必须提供用户信息，并且必须保障所需金额的客户价值。当预付费工具停止使用时，发行人将根据《支付服务法》进行退款流程。

解析规则为：

- **封闭式或商家使用的储值**倾向于预付费分析。
- **人与人之间的可转让价值**倾向于资金转移分析。
- **等值现金的兑换或广泛的可转让性**应该引发更高风险的法律审查，而不是简单的“积分”或“预付”标签。

对于注册的第三方预付费发行人群体，请使用 [[payments/prepaid-payment-instrument-issuers-japan-index]] 而不是创建长尾发行人页面。

## 产品分解

| 产品表面 | 可能的法律层面 |
|---|---|
| 代码支付应用 | 转账余额、预付余额、信用卡、银行借记卡、积分、优惠券（例如[[JapanFG/paypay|PayPay]]、[[JapanFG/kyash|Kyash]]）。 |
| 零售电子货币 | 预付支付工具、忠诚度积分、商户结算合同。 |
| 市场钱包 | 用于卖家付款的资金转账余额、用于购买的预付积分、托管/结算功能（比较 [[JapanFG/merpay|Merpay]]）。 |
| 游戏/应用积分 | 基于服务器的预付费支付工具、促销积分、平台费用经济。 |
| 稳定币 / EPI 桥 | 电子支付工具、资金转移路径、信任路径、加密资产/ECISB 边界（参见 [[fintech/japan-ecisb-license|Japan ECISB license]]）。 |

## 为什么它很重要

- **消费者保护**：退款、保障、披露和滥用控制取决于法律层面。
- **商业经济学**：预付金额、汇款余额、奖励积分和商户结算都会创建不同的负债和浮动配置文件。
- **许可策略**：公司可以避免 P2P 转账以保持类似预付费的结构，或者获得资金转账注册以实现钱包网络效应。
- **JapanFG 路由**：银行集团、支付集团、零售商和稳定币项目应通过法定平衡层进行比较，而不是通过应用程序 UI 进行比较。
